# Skill - Pre-Commit Knowledge Quality Gate

## Purpose
Run a repeatable pre-commit quality gate for Max OS changes so Markdown structure, links, whitespace, generated byproducts, and public-template privacy risks are checked before work is committed or proposed upstream.

## When to Use
Use this skill:
- before committing changes to Max OS;
- before showing a public repo diff for review;
- after adding or updating skills, workflows, templates, policies, or operating rules;
- after processing many inbox files into canonical notes;
- before propagating private workspace learnings into the public template.

## Trigger Phrases
- "run pre-commit quality gate"
- "check before commit"
- "is this safe to commit"
- "run the Max OS quality gate"
- "validate this branch before pushing"

## Inputs
- Workspace root
- Optional mode: changed files or full workspace
- Optional public-template privacy mode
- Optional denylist file for private names, clients, projects, or terms

## Outputs
- Pass/fail summary for each gate
- Knowledge lint summary
- Whitespace diff check result
- Runtime byproduct findings
- Optional public-template privacy and secret-scan findings
- Recommended fixes before commit

## Default Commands

Ensure local setup once per clone, and rerun at session start:

```bash
sh 15_Skills/tools/ensure_local_setup.sh
```

Changed-file gate:

```bash
python3 15_Skills/tools/maxos_quality_gate.py --root .
```

Full workspace gate:

```bash
python3 15_Skills/tools/maxos_quality_gate.py --root . --full
```

Public-template gate:

```bash
python3 15_Skills/tools/maxos_quality_gate.py --root . --full --public-template
```

Public-template gate with a private denylist:

```bash
python3 15_Skills/tools/maxos_quality_gate.py --root . --full --public-template --denylist 00_System/private_public_denylist.txt
```

Recommended local denylist setup:

```bash
cp 00_System/public_template_denylist.example.txt .maxos/public_template_denylist.txt
```

Then edit `.maxos/public_template_denylist.txt` with private names, clients, projects, and terms that should never appear in public-template changes. The quality gate automatically uses that local file during `--public-template` scans when it exists.

## Gates

## Local Git Hook
Git normally looks for executable hook scripts in `.git/hooks/`, but that folder is not versioned. Max OS stores the shared hook in `.githooks/pre-commit` and installs it with:

```bash
git config core.hooksPath .githooks
```

`sh 15_Skills/tools/ensure_local_setup.sh` checks that setting, repairs it when needed, and writes clone-local status to `.maxos/local_setup_status.yaml`. Missing status or `ready: false` means the clone is not initialized.

After `core.hooksPath=.githooks` is applied in a clone, every `git commit` runs `.githooks/pre-commit` before the commit is created. If the script exits with a non-zero status, Git aborts the commit. Fix the issue, stage the fix, and commit again.

The hook can be bypassed with `git commit --no-verify` or `MAXOS_SKIP_QUALITY_GATE=1 git commit ...`, but that should be treated as an explicit exception.

### Git Diff Whitespace
Runs:

```bash
git diff --check
```

This catches trailing whitespace, conflict markers, and common patch hygiene issues.

### Knowledge Lint
Runs [[Skill - Knowledge System Lint and Link Check]] through:

```bash
python3 15_Skills/tools/knowledge_lint.py --root . --changed-only --fail-on error
```

For full-template review, use the full workspace mode.

### Runtime Byproducts
Fails if tracked or untracked runtime byproducts are present:
- `__pycache__/`
- `*.pyc`
- `*.pyo`
- `*.pyd`

These should be ignored or removed, not committed.

### Public-Template Privacy and Secret Scan
When `--public-template` is used, scan changed or full text files for:
- obvious secret/credential patterns;
- optional denylist terms supplied by `.maxos/public_template_denylist.txt` or `--denylist`;
- private terms that should not enter the public template.

This is a heuristic gate, not a replacement for human review.

## Process
1. Run `git status --short --branch`.
2. Run `sh 15_Skills/tools/ensure_local_setup.sh`.
3. Confirm `.maxos/local_setup_status.yaml` says `ready: true`.
4. Run the changed-file quality gate manually when preparing a review.
5. If preparing public repo changes, run the public-template gate with a private denylist when available.
6. Fix failures before commit.
7. If warnings or privacy findings are false positives, document why before continuing.
8. Show `git diff --stat` and a concise summary before asking for commit approval.

## Quality Checks
- [ ] `git diff --check` passes
- [ ] knowledge lint passes
- [ ] no runtime byproducts are tracked or present
- [ ] public-template privacy scan passes when applicable
- [ ] untracked files intended for the commit are listed
- [ ] generated files are intentionally included or ignored
- [ ] no private workspace content is copied into public templates

## Anti-Patterns
- Committing before checking untracked files.
- Running only text review when deterministic checks exist.
- Treating a secret/privacy scan as complete proof that public content is safe.
- Ignoring untracked generated files because `git diff` does not show them.
- Assuming Git hooks are automatically installed after clone; each clone must run the installer or set `core.hooksPath`.
- Pushing to a remote without explicit approval.

## Final Reporting Format
Report:
- branch and status;
- command run;
- pass/fail result for each gate;
- fixes applied;
- remaining risks or manual review items;
- whether the branch is ready to commit, not whether it is approved to push.
